# 生成AIツール診断サイト 仕様書（Claude Code向け / Netlify無料公開前提）

## 1. 目的

企業の担当者が、短時間の入力で自社に合いやすい生成AIツールを診断できる公開Webページを提供する。

本システムは以下の2軸で診断する。

- A. ツール適合性診断
    
- B. 導入準備度診断
    

対象ユーザーは生成AIの導入をまだ迷っている潜在層であり、重い入力を避け、短時間で完了できることを重視する。

---

## 2. 前提条件

- 公開方法: Netlify Free で公開
    
- コスト前提: 実質 API 利用料のみ
    
- バックエンド: Netlify Functions を使用
    
- データ保存: 原則しない
    
- DB: 使用しない
    
- 比較対象ツール情報: リポジトリ内の JSON / TypeScript 定数として事前格納
    
- 診断結果: 画面表示のみ。永続保存しない
    
- スパム対策: 実装しない
    
- ログイン機能: 実装しない
    
- PDF出力: 実装しない
    
- メール送信: 実装しない

## 3. システム構成

### フロントエンド

- Next.js（App Router）
    
- TypeScript
    
- Tailwind CSS
    
- 1ページ完結の診断UI
    

### バックエンド

- Netlify Functions
    
- API キーは Netlify の環境変数に保存
    

### AI利用方針

LLM は以下の用途に限定する。

- 自社サイトURLから取得した内容の要約
    
- 自由記述の要約
    
- ルールエンジンが出した結果の自然文レポート化
    

LLM に以下は任せない。

- 最終スコア計算
    
- 推薦順位の決定
    
- 各ツールの知識検索
    

スコア計算はローカルのルールベースで行う。

## 7. URL解析要件

### 目的

入力された自社サイトURLから会社理解の補助情報を得る。

### 方針

- URL解析は補助機能であり、診断の主判定にはしない
    
- 取得失敗しても診断を継続できること
    
- トップページの本文テキストを取得して要約する
    
- 要約結果は Q4 の事業内容の初期値として利用可能にする
    

### 実装方針

- クライアントから URL を送信
    
- Netlify Function で対象 HTML を取得
    
- HTML から本文テキストを抽出
    
- LLM に短く要約させる
    
- 取得失敗時はスキップ

## 8. 診断ロジック要件

### 8.1 診断の基本方針

診断は以下の2本立てで実装する。

- A. ツール適合性診断
    
- B. 導入準備度診断
    

### 8.2 ツール適合性スコア

各ツールに対し 0〜100 点を算出する。

配点:

- 普段の社内ツールとの相性: 35点
    
- 効率化したい業務との相性: 30点
    
- 情報の扱い方との相性: 15点
    
- 導入の始めやすさ: 10点
    
- 予算との相性: 10点
    

### 8.3 導入準備度スコア

0〜100 点を算出する。

配点:

- AI利用経験: 25点
    
- 推進担当の有無: 20点
    
- 導入範囲の明確さ: 20点
    
- 情報の扱い方の整理度: 20点
    
- 回答の具体性: 15点
    

### 8.4 診断信頼度

- 高 / 中 / 低 の3段階
    
- 入力項目数・URL解析可否・自由記述有無から算出

## 9. スコアリングルール

### 9.1 一次判定

- Microsoft系が中心 → Copilot を強く加点
    
- Google系が中心 → Gemini を強く加点
    
- 両方使っている / 特に決まっていない / その他 → ChatGPT を基準候補として加点
    

### 9.2 用途補正

- 提案書・文章作成・アイデア出し → ChatGPT 加点
    
- 情報収集・資料整理 → Gemini 加点
    
- Excel作業・PowerPoint・Word 業務 → Copilot 加点
    
- 会議メモ・要約 → 全ツール加点。ただし環境親和性で差をつける
    

### 9.3 導入しやすさ補正

- まずは個人利用から → ChatGPT 加点
    
- 一部部署で試したい → 全ツール加点
    
- 全社で検討したい + Microsoft中心 → Copilot 加点
    
- 全社で検討したい + Google中心 → Gemini 加点
    

### 9.4 情報の扱い方による補正

- 公開情報中心 → 全ツール減点なし
    
- 社内資料を扱う → 管理性の低い運用想定に軽い注意
    
- 顧客情報や社外秘も扱う → 全ツールに注意フラグ付与
    
- かなり慎重に扱う必要がある → ツール推薦は出すが、結果に「事前整理推奨」を明示
    

### 9.5 準備度ペナルティ

以下に該当する場合、導入準備度を下げる。

- まだほとんど使っていない
    
- 推進担当がいない
    
- 全社導入したいが利用範囲が曖昧
    
- 慎重な情報を扱うがルールが未定

## 11. レポート仕様

### 必須表示項目

1. 最も合いやすいツール
    
2. 3ツール比較
    
3. まず効きやすい業務 3件
    
4. 導入準備度
    
5. はじめの一歩
    
6. 注意点
    
7. 診断の見方（断定ではなく目安である旨）
    

### 表示内容詳細

#### 11.1 最上部サマリー

- 推薦1位ツール名
    
- 相性スコア
    
- 一言コメント
    

#### 11.2 比較ブロック

- ChatGPT / Gemini / Copilot の3列比較
    
- 各スコア
    
- 向いているケース
    
- 注意点
    

#### 11.3 まず効きやすい業務

- 入力内容をもとに3件抽出
    
- 例: 提案書のたたき台作成、会議メモ要約、情報収集と整理
    

#### 11.4 導入準備度

- 80〜100: すぐ試しやすい
    
- 60〜79: 一部業務からなら始めやすい
    
- 40〜59: 先に対象業務の整理が必要
    
- 0〜39: まずは社内方針や進め方の整理が必要
    

#### 11.5 はじめの一歩

- 最初に試す業務
    
- 最初に試す人数
    
- 最初に決めるルール
    
- 相談時に確認したい論点
    

### トーン要件

- 断定しすぎない
    
- 「〜の可能性があります」「まずは〜がおすすめです」を基本とする
    
- セキュリティについては「安全」と断定せず「扱う情報に応じて整理が必要」と表現する

## 15. Claude Codeへの依頼方針

Claude Code には以下の順番で依頼すること。

1. この仕様書を読み込ませる
    
2. MVPとして必要な機能をタスク分解させる
    
3. ファイル構成を提案させる
    
4. まず静的データで診断ロジックを実装させる
    
5. 次にフォームUIを実装させる
    
6. 最後に Netlify Functions と API 接続を実装させる
    

### Claude Code に伝える注意点

- LLM は最終判断に使わない
    
- スコアリングは関数で明示的に実装する
    
- 静的データを書き換えやすい構造にする
    
- データ保存はしない
    
- 実装は MVP 優先でシンプルにする
    
- Netlify Free で動く前提で余計な依存を増やさない
