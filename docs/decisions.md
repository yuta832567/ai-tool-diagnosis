# Decisions

## 2026-04-08
### 目的
潜在層の顧客が、短時間の入力で自社に合いやすい生成AIツールを把握できる診断サイトを作る。

### 今回の方針
- 診断は2軸で行う
  - A. ツール適合性診断
  - B. 導入準備度診断
- ヒアリングは軽めにする
- 初心者でも意味が分かる言葉を使う
- 専門用語はできるだけ避ける
- 診断は断定ではなく「相性」「適合度」として返す

### 技術方針
- 公開先は Netlify Free
- 実装は Next.js + TypeScript + Tailwind CSS
- バックエンドは Netlify Functions
- データベースは使わない
- 比較対象ツールの情報は静的データとして事前に持つ
- 診断結果は保存しない
- PDF出力はしない
- メール送信はしない
- スパム対策は入れない

### AIの使い方
- LLMに任せるもの
  - URLから取得した会社情報の要約
  - 自由記述の要約
  - 診断結果の自然文レポート化
- LLMに任せないもの
  - 最終スコア計算
  - 順位決定
  - ツール知識の都度検索

### 比較対象
- ChatGPT
- Microsoft 365 Copilot
- Google Gemini

### 診断UI方針
- 3ステップ構成
- 60〜90秒で終わることを目指す
- 選択式中心
- その他を必ず用意する
- 必要に応じて自由記述欄を出す

### 補足
- URL解析は補助機能であり、主判定には使いすぎない
- スコアリングはルールベースで実装する
- 将来的にツール追加しやすい構造にする

## 2026-04-08（実装フェーズ追加決定）

### ディレクトリ構成の決定
- `src/` フォルダは作らず、`app/` と並列に `components/`, `lib/`, `data/` を配置する
- Next.js App Router の `app/` は `app/app/` に存在（プロジェクトルートが `app/`）
- `@/*` は `./` (プロジェクトルート) に解決される（tsconfig の paths 設定に従う）

### スコアリングの実装方針（確定）
- スコアリングは完全にクライアントサイドのルールベース関数で実装
- API呼び出しなしで即時計算する（MVP段階）
- 配点は仕様書 §8.2 の通り（環境35点・業務30点・情報15点・始めやすさ10点・予算10点）
- 5軸スコアは表示用に各軸を 0〜100 に正規化して保持する

### 結果画面のデザイン方針（リデザイン決定）
- 「診断結果」から「簡易提案レポート」の体裁に変更
- 対象: 法人意思決定者・担当者が見ることを想定したビジネス向けUI
- 構成:
  - I. エグゼクティブサマリー（推奨ツール・導入スタンス・推薦理由3つ）
  - II. ツール適合性評価（横棒グラフ + 3ツール比較表）
  - III. 5軸分析（既存ツール相性・業務相性・始めやすさ・情報の扱いやすさ・費用感）
  - IV. 活用が期待できる業務（3枚のカード、各カードに期待効果・なぜ合うか・最初の試し方）
  - V. 導入準備度（スコア定義文・現状評価・次にやること3つ）
  - VI. 総合所見（なぜ/どう始めるか/注意点）
  - CTA（相談ボタン・やり直しボタン）
- レーダーチャートは不採用（スマホ対応が難しいため）→ 横棒グラフに統一
- グラデーションは最小限に抑える
- 丸み・余白・タイポグラフィ階層・罫線でビジネス感を演出

### 型設計の方針
- `DiagnosisResult` を拡張し、UIに必要な全データを一括返却する設計とする
- 後方互換フィールド（`readinessScore`, `readinessLabel`, `recommendedUsecases`, `nextSteps`）を残す
- LLM連携後もこの型を維持し、テキストフィールドだけ LLM 生成に差し替える

### 保留・未実装
- Netlify Functions（analyze API）は未実装。現状はクライアントのみで完結
- URL解析・LLM連携は Phase 6 で実装予定
- `data/readinessRules.json` は scoring.ts にインライン実装した（ファイル分割は将来対応）

## 2026-04-09（PDF出力方針・結果画面UI強化）

### PDF出力の方針（確定）
- 実装方式：**印刷用レイアウト + ブラウザPDF保存**（外部ライブラリなし）
- `window.print()` をトリガーするボタンを結果画面に設置
- ボタン名：「レポートを保存」
- ボタン押下後、ツールチップで「印刷画面で『PDFに保存』を選択してください」と案内
- `@media print` スタイル（globals.css）で印刷版レイアウトを定義
  - 不要要素（CTA・再診断ボタン・ヘッダー操作系）は `print:hidden` で非表示
  - 白背景・黒文字で可読性優先（`print-color-adjust: exact` でカラー維持）
  - セクションが途中で分割されないよう `print:break-inside-avoid` を設定
  - 印刷専用ヘッダー（タイトル・診断日・会社名）を `hidden print:block` で表示
- 比較テーブルはスマホで横スクロール可能（`overflow-x-auto` ラッパー + テーブル `min-w`）

### 不採用方針（PDF）
- `react-pdf` / `jspdf` 等のライブラリ：Netlify Free バンドルサイズ・依存増加を避ける
- サーバーサイドPDF生成：データ保存なし方針と相性が悪い

### 結果画面UIの強化方針（確定）
- ツール比較：カード3枚 → 横スクロール対応の3列比較テーブルに変更
- 導入準備度：横棒グラフ → SVGゲージ（円形）+ スコア・総評を横並びに変更
- セクションヘッダー：数字バッジ（黒丸）+ 大文字ラベルに統一
- 総合所見：左サイドボーダー付きのブロック引用スタイルに変更
- 「レポートを保存」ボタンをヘッダー部とCTA部の両方に配置
- 印刷専用ヘッダー（`hidden print:block`）で会社名・診断日・信頼度を表示
- `companyName` を `DiagnosisResult` コンポーネントの Props に追加、`DiagnosisForm` から受け渡し
